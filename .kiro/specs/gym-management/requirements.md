# Requirements Document

## Introduction

A production-ready Gym Management Web Application built with Django and Django REST Framework, using server-rendered templates with Tailwind CSS. The system uses facial recognition (face-api.js, browser-based) as the primary attendance tracking method, and supports member management, billing, role-based access control, and reporting. The application targets gym administrators and staff who need a reliable, modern tool to manage day-to-day gym operations.

## Glossary

- **System**: The Gym Management Web Application
- **Admin**: A user with full access to all modules and settings
- **Staff**: A user with access to attendance tracking and member handling only
- **Member**: A registered gym member with a profile, membership plan, and face descriptor
- **Face_Descriptor**: A numerical vector (128-float array) generated by face-api.js representing a member's facial features
- **FaceMatcher**: The face-api.js utility that compares a detected face descriptor against stored descriptors
- **Membership_Plan**: A billing plan with a name, price, and duration in days
- **Payment**: A record of a financial transaction linked to a member and a membership plan period
- **Attendance**: A record of a member's check-in event on a given date
- **Dashboard**: The main overview page showing key gym metrics
- **face-api.js**: A browser-based JavaScript library for face detection and recognition
- **SSD_MobileNetV1**: A face detection model used by face-api.js
- **FaceLandmark68Net**: A face landmark detection model used by face-api.js
- **FaceRecognitionNet**: A face recognition model used by face-api.js

## Requirements

### Requirement 1: User Authentication and Role-Based Access Control

**User Story:** As a gym owner, I want role-based access control so that admins have full system access while staff are limited to attendance and member handling.

#### Acceptance Criteria

1. WHEN a user submits valid credentials, THE System SHALL authenticate the user and redirect them to the dashboard.
2. WHEN a user submits invalid credentials, THE System SHALL display an error message and remain on the login page.
3. WHEN an authenticated Admin navigates to any module, THE System SHALL grant access to all modules including billing, settings, and user management.
4. WHEN an authenticated Staff user attempts to access billing or admin-only settings, THE System SHALL deny access and display a permission error.
5. WHEN an unauthenticated user attempts to access any protected page, THE System SHALL redirect them to the login page.
6. THE System SHALL support exactly two roles: Admin and Staff.
7. WHEN a user logs out, THE System SHALL invalidate the session and redirect to the login page.

---

### Requirement 2: Member Registration with Face Capture

**User Story:** As a staff member, I want to register new gym members with their photo and face descriptor so that they can be identified by the facial recognition system during attendance.

#### Acceptance Criteria

1. WHEN a staff member opens the member registration form, THE System SHALL activate the webcam using `getUserMedia()` and display a live video feed.
2. WHEN a face is detected in the webcam feed, THE System SHALL use face-api.js to extract a 128-float face descriptor vector.
3. WHEN a face descriptor is successfully extracted, THE System SHALL store the descriptor as a JSON array in the `face_descriptor` field of the Member record.
4. IF no face is detected during registration, THEN THE System SHALL display an error message and prevent form submission.
5. THE System SHALL require the following fields for member registration: `full_name`, `phone`, `email`, `photo`, `face_descriptor`, `membership_plan`, and `join_date`.
6. WHEN a member is saved, THE System SHALL automatically calculate and store the `expiry_date` based on `join_date` and the selected `Membership_Plan`'s `duration_days`.
7. WHEN a member is saved, THE System SHALL set the member's `status` to "Active" if `expiry_date` is in the future, or "Expired" otherwise.
8. IF a duplicate email is submitted during registration, THEN THE System SHALL display a validation error and prevent saving.

---

### Requirement 3: Member Management

**User Story:** As an admin, I want to view, edit, and manage all gym members so that I can keep member records accurate and up to date.

#### Acceptance Criteria

1. THE System SHALL display a paginated list of all members with their name, status, membership plan, and expiry date.
2. WHEN an admin searches for a member by name or email, THE System SHALL return matching members in the list.
3. WHEN an admin edits a member's profile, THE System SHALL update the record and reflect changes immediately in the member list.
4. WHEN an admin deletes a member, THE System SHALL remove the member record and all associated attendance records.
5. WHEN an admin views a member's detail page, THE System SHALL display the member's full profile including payment history and attendance history.
6. THE System SHALL display each member's current status as "Active", "Expired", or "Suspended".

---

### Requirement 4: Membership Plans

**User Story:** As an admin, I want to create and manage membership plans so that I can offer different pricing tiers to members.

#### Acceptance Criteria

1. THE System SHALL allow admins to create a `Membership_Plan` with a `name`, `price`, and `duration_days`.
2. WHEN a membership plan is updated, THE System SHALL reflect the new details on all future payments and registrations.
3. WHEN an admin attempts to delete a membership plan that has active members assigned to it, THE System SHALL prevent deletion and display a warning.
4. THE System SHALL display all available membership plans in a list with their name, price, and duration.

---

### Requirement 5: Billing and Payments

**User Story:** As an admin, I want to record payments and track billing so that I can manage revenue and member subscription status.

#### Acceptance Criteria

1. WHEN a payment is recorded for a member, THE System SHALL create a `Payment` record with `member`, `amount`, `date_paid`, `period_start`, `period_end`, `payment_method`, and optional `notes`.
2. WHEN a payment is recorded, THE System SHALL update the member's `expiry_date` to the `period_end` of the new payment.
3. WHEN a payment is recorded, THE System SHALL update the member's `status` to "Active" if the new `expiry_date` is in the future.
4. THE System SHALL display a list of all payments filterable by member and date range.
5. WHEN a member's `expiry_date` is in the past, THE System SHALL mark that member as overdue and display them in an overdue members list.
6. THE System SHALL display the total monthly revenue on the dashboard, calculated as the sum of all payments with `date_paid` in the current calendar month.

---

### Requirement 6: Facial Recognition Attendance

**User Story:** As a staff member, I want to use facial recognition to check in members automatically so that attendance tracking is fast and contactless.

#### Acceptance Criteria

1. WHEN the attendance page is opened, THE System SHALL load all stored face descriptors from the backend via a single API call and cache them in the browser.
2. WHEN the attendance page is active, THE System SHALL open a live webcam feed and continuously run face detection at a maximum frequency of once per second.
3. WHEN a face is detected in the live feed, THE System SHALL use `FaceMatcher` to compare the detected descriptor against all cached member descriptors.
4. WHEN a match is found with a distance below the configured threshold, THE System SHALL send a POST request to `/attendance/checkin/` with the matched member's ID.
5. WHEN a successful check-in POST is accepted by the backend, THE System SHALL display a success message: "✅ Welcome, [Member Name]".
6. WHEN no match is found for a detected face, THE System SHALL display "❌ Face not recognized" and offer a manual check-in option.
7. WHEN multiple faces are detected simultaneously, THE System SHALL process each face independently and match each to the best candidate.
8. IF a member has already checked in on the current date, THEN THE System SHALL reject the duplicate check-in request and display an appropriate message.
9. WHEN the backend receives a check-in request, THE System SHALL create an `Attendance` record with `member`, `check_in_time`, `date`, and `method` set to "Face".
10. WHERE manual check-in is used, THE System SHALL create an `Attendance` record with `method` set to "Manual".

---

### Requirement 7: Face Recognition Model Loading

**User Story:** As a system operator, I want face-api.js models to be hosted locally so that the application works without external CDN dependencies at runtime.

#### Acceptance Criteria

1. THE System SHALL serve the `ssdMobilenetv1`, `faceLandmark68Net`, and `faceRecognitionNet` model files from a local static directory.
2. WHEN the attendance or registration page loads, THE System SHALL load all three models from the local static path before activating the webcam.
3. IF any model fails to load, THEN THE System SHALL display an error message and disable the facial recognition feature gracefully.
4. WHEN all models are loaded successfully, THE System SHALL display a ready indicator before starting the webcam feed.

---

### Requirement 8: Dashboard and Reporting

**User Story:** As an admin, I want a dashboard with key metrics so that I can monitor gym performance at a glance.

#### Acceptance Criteria

1. THE Dashboard SHALL display the total number of registered members.
2. THE Dashboard SHALL display the number of currently active members (status = "Active").
3. THE Dashboard SHALL display the total number of check-ins recorded for the current calendar day.
4. THE Dashboard SHALL display the total revenue for the current calendar month.
5. WHEN an admin requests an attendance CSV export, THE System SHALL generate and return a CSV file containing all attendance records with member name, date, check-in time, and method.
6. WHERE a date range filter is applied to the CSV export, THE System SHALL include only attendance records within the specified range.

---

### Requirement 9: UI/UX and Layout

**User Story:** As a user, I want a modern, responsive interface so that I can use the system comfortably on both desktop and mobile devices.

#### Acceptance Criteria

1. THE System SHALL render all pages using a sidebar and top navbar layout.
2. THE System SHALL use Tailwind CSS for all styling.
3. WHEN an action succeeds or fails, THE System SHALL display a toast notification with appropriate feedback.
4. THE System SHALL be fully usable on screen widths from 375px (mobile) to 1920px (desktop).
5. WHEN a page is loading data, THE System SHALL display a loading indicator to the user.

---

### Requirement 10: Deployment and Configuration

**User Story:** As a developer, I want the application to be deployable to a cloud platform with environment-based configuration so that it can run in both development and production environments.

#### Acceptance Criteria

1. THE System SHALL use SQLite as the database in development and PostgreSQL in production, controlled by an environment variable.
2. THE System SHALL serve static files using WhiteNoise in production.
3. THE System SHALL read all sensitive configuration values (secret key, database URL, allowed hosts) from environment variables.
4. THE System SHALL be deployable to Railway or Render without manual server configuration.
5. WHEN `DEBUG=False`, THE System SHALL not expose stack traces or internal error details to end users.
